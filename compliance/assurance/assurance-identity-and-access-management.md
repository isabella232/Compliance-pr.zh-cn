<span data-ttu-id="66d16-109">工程师被分配到服务团队以支持生产服务通过标识管理工具请求服务团队帐户的资格 (IDM) 。</span><span class="sxs-lookup"><span data-stu-id="66d16-109">Engineers assigned to a service team to support production services request eligibility for a service team account through the Identity Management Tool (IDM).</span></span> <span data-ttu-id="66d16-110">资格申请触发了一系列人员检查，以确保工程师已通过所有云筛选要求、完成必要的培训，并获得了帐户创建前的适当管理批准。</span><span class="sxs-lookup"><span data-stu-id="66d16-110">The request for eligibility triggers a series of personnel checks to ensure the engineer has passed all cloud screening requirements, completed necessary training, and received appropriate management approval prior to account creation.</span></span> <span data-ttu-id="66d16-111">只有在满足所有资格要求后，才能为请求的环境创建服务团队帐户。</span><span class="sxs-lookup"><span data-stu-id="66d16-111">Only after meeting all eligibility requirements can a service team account be created for the requested environment.</span></span> <span data-ttu-id="66d16-112">若要维护服务团队帐户的资格，人员每年必须通过基于角色的培训，每两年重新进行一次审查。</span><span class="sxs-lookup"><span data-stu-id="66d16-112">To maintain eligibility for a service team account, personnel must go through role-based training annually and rescreening every two years.</span></span> <span data-ttu-id="66d16-113">未能完成或通过这些检查会导致资格被自动吊销。</span><span class="sxs-lookup"><span data-stu-id="66d16-113">Failure to complete or pass these checks result in eligibilities automatically being revoked.</span></span>

<span data-ttu-id="66d16-114">服务团队帐户不会授予任何长期管理员权限或客户内容的访问权限。</span><span class="sxs-lookup"><span data-stu-id="66d16-114">Service team accounts do not grant any standing administrator privileges or access to customer content.</span></span> <span data-ttu-id="66d16-115">当工程师需要其他访问权限来支持其 Microsoft 365 服务时，他们使用名为"密码箱"的访问管理工具请求临时提升的资源访问权限。</span><span class="sxs-lookup"><span data-stu-id="66d16-115">When an engineer requires additional access to support their Microsoft 365 service, they request temporary elevated access to the resources they require using an access management tool called Lockbox.</span></span> <span data-ttu-id="66d16-116">密码箱将提升的访问权限限制为完成分配的任务所需的最小特权、资源和时间。</span><span class="sxs-lookup"><span data-stu-id="66d16-116">Lockbox restricts elevated access to the minimum privileges, resources, and time needed to complete the assigned task.</span></span> <span data-ttu-id="66d16-117">如果授权审阅者批准 JIT 访问请求，则只会向工程师授予一个临时帐户，该帐户仅具有完成其分配的工作所需的权限。</span><span class="sxs-lookup"><span data-stu-id="66d16-117">If an authorized reviewer approves the JIT access request, the engineer is granted a temporary account with only the privileges necessary to complete their assigned work.</span></span> <span data-ttu-id="66d16-118">此临时帐户需要多重身份验证，在批准期限到期后自动删除。</span><span class="sxs-lookup"><span data-stu-id="66d16-118">This temporary account requires multi-factor authentication and is automatically deleted after the approved period expires.</span></span>

<span data-ttu-id="66d16-119">JEA 由 IDM 资格和密码箱角色在请求 JIT 访问时强制执行。</span><span class="sxs-lookup"><span data-stu-id="66d16-119">JEA is enforced by IDM eligibilities and Lockbox roles at the time of request for JIT access.</span></span> <span data-ttu-id="66d16-120">只接受对工程师资格范围内资产的访问请求，并传递给审批者。</span><span class="sxs-lookup"><span data-stu-id="66d16-120">Only requests for access to assets within the scope of the engineer's eligibilities are accepted and passed on to the approver.</span></span> <span data-ttu-id="66d16-121">密码箱会自动拒绝超出工程师资格和密码箱角色范围的 JIT 请求，包括超出允许阈值的请求。</span><span class="sxs-lookup"><span data-stu-id="66d16-121">Lockbox automatically rejects JIT requests that are outside the scope of the engineer's eligibilities and Lockbox roles, including requests that exceed allowed thresholds.</span></span>  

## <a name="how-does-microsoft-365-use-role-based-access-control-rbac-with-lockbox-to-enforce-least-privilege"></a><span data-ttu-id="66d16-122">如何使用Microsoft 365 RBAC (密码) 基于角色的访问控制来强制实施最小特权？</span><span class="sxs-lookup"><span data-stu-id="66d16-122">How does Microsoft 365 use role-based access control (RBAC) with Lockbox to enforce least privilege?</span></span>

<span data-ttu-id="66d16-123">服务团队帐户不会授予任何长期管理员权限或客户内容的访问权限。</span><span class="sxs-lookup"><span data-stu-id="66d16-123">Service team accounts do not grant any standing administrator privileges or access to customer content.</span></span> <span data-ttu-id="66d16-124">受限管理员权限的 JIT 请求通过密码箱进行管理。</span><span class="sxs-lookup"><span data-stu-id="66d16-124">JIT requests for limited administrator privileges are managed through Lockbox.</span></span> <span data-ttu-id="66d16-125">密码箱使用 RBAC 来限制 JIT 提升请求工程师可以提出的类型，从而提供额外的保护层来强制实施最小特权。</span><span class="sxs-lookup"><span data-stu-id="66d16-125">Lockbox uses RBAC to limit the types of JIT elevation requests engineers can make, providing an additional layer of protection to enforce least privilege.</span></span> <span data-ttu-id="66d16-126">RBAC 还通过将服务团队帐户限制到适当的角色来帮助强制实施职责分离。</span><span class="sxs-lookup"><span data-stu-id="66d16-126">RBAC also helps enforce separation of duties by limiting service team accounts to appropriate roles.</span></span>
<span data-ttu-id="66d16-127">支持服务的工程师将基于其角色被授予安全组的成员身份。</span><span class="sxs-lookup"><span data-stu-id="66d16-127">Engineers supporting a service are granted membership to security groups based on their role.</span></span> <span data-ttu-id="66d16-128">安全组的成员身份不会授予任何特权访问权限。</span><span class="sxs-lookup"><span data-stu-id="66d16-128">Membership in a security group does not grant any privileged access.</span></span> <span data-ttu-id="66d16-129">相反，安全组允许工程师使用密码箱请求 JIT 提升（如果需要支持系统）。</span><span class="sxs-lookup"><span data-stu-id="66d16-129">Instead, security groups allow engineers to use Lockbox to request JIT elevation when required for supporting the system.</span></span> <span data-ttu-id="66d16-130">工程师可以提出的特定 JIT 请求受其安全组成员身份限制。</span><span class="sxs-lookup"><span data-stu-id="66d16-130">The specific JIT requests an engineer can make are limited by their security group memberships.</span></span>

## <a name="how-does-microsoft-365-handle-remote-access-to-production-systems"></a><span data-ttu-id="66d16-131">如何Microsoft 365对生产系统的远程访问？</span><span class="sxs-lookup"><span data-stu-id="66d16-131">How does Microsoft 365 handle remote access to production systems?</span></span>

<span data-ttu-id="66d16-132">Microsoft 365组件位于地理位置上与运营团队分开的数据中心中。</span><span class="sxs-lookup"><span data-stu-id="66d16-132">Microsoft 365 system components are housed in datacenters geographically separated from the operations teams.</span></span> <span data-ttu-id="66d16-133">数据中心人员对系统没有逻辑Microsoft 365访问权限。</span><span class="sxs-lookup"><span data-stu-id="66d16-133">Datacenter personnel do not have logical access to Microsoft 365 systems.</span></span> <span data-ttu-id="66d16-134">因此，Microsoft 365团队人员通过远程访问来管理环境。</span><span class="sxs-lookup"><span data-stu-id="66d16-134">As a result, Microsoft 365 service team personnel manage the environment through remote access.</span></span> <span data-ttu-id="66d16-135">只有经过授权经理批准后，才Microsoft 365远程访问以支持远程访问的服务团队人员。</span><span class="sxs-lookup"><span data-stu-id="66d16-135">Service team personnel who require remote access to support Microsoft 365 are only granted remote access after approval from an authorized manager.</span></span> <span data-ttu-id="66d16-136">所有远程访问都使用 FIPS 140-2 兼容 TLS 进行安全远程连接。</span><span class="sxs-lookup"><span data-stu-id="66d16-136">All remote access uses FIPS 140-2 compatible TLS for secure remote connections.</span></span>
